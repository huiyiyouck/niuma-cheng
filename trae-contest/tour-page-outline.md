## 设计理念（F1 分层叙事落地）

**为什么分层**：工作流层（4 角色 + PRD→Review→部署门禁）与 TRAE 官方角色智能体（Backend Architect 等）有同质化风险。真正的差异化卖点是 P14 后的组织架构层——指挥官—参谋长制、公告板、BCR 闭环。但组织架构层太抽象，评委 60 秒看不懂。所以**先用工作流层抓住注意力（配智能体链接立即可试），再用组织架构层压轴拉开差异（配三个真实上线项目为活证据）**。

**两层之间的桥**：6.8 交叉引用——小报一个功能用该智能体跑一轮，是"工作流产出现实产品"的活证据，连接两层。

### 第一层：工作流层（前 60 秒，立即可试）

> 目标：评委 60 秒看懂"这是一套有门禁的 AI 工作流"，并能立刻点链接试用。

#### 1.1 核心概念：4 角色 + 门禁

**内容**：
默认是 General（通用助手），不擅自进入工作流。说"你是 PM"才切换角色、加载规则。4 个角色各有职责边界，不越权。

**4 角色卡片**（沿用 proposal 的 participants-grid 风格）：
| 角色 | 职责 | 不做 |
|---|---|---|
| 🎯 PM | 需求、PRD、界面要点、验收标准 | 不做技术选型、不写代码 |
| 🏗️ Architect | 技术方案、架构设计 | 不做产品决策、不写业务代码 |
| 💻 Developer | 实现、自测、Bugfix | 不自行决定需求范围 |
| 🚀 DevOps | 部署、运维、发布 | 不做产品/技术决策 |

**门禁流程图**（新增可视化）：
```
PRD 创建 → 待 Review → Review 通过 → 定稿 → 启动迭代 → 开发 → 部署门禁
   ↑                                    ↑              ↑
 门禁：不审不能定稿              门禁：无PRD不开工   门禁：不自测不部署
```

> 可视化标注：横向流程图，3 个门禁点用红色🚫图标标注，配文字说明每个门禁拦什么

#### 1.2 评委可立即试

**内容**：
点下面的链接，一键导入 TRAE IDE，5 分钟跑完一轮"给 TODO 功能写 PRD 并走 Review"。你会亲手触发门禁——试着跳过 Review 直接开发，看 AI 拒不拒绝你。

**CTA**：「导入智能体到 TRAE」→ 智能体分享链接（待 1.5 回填）

**体验脚本速览**（折叠区，点击展开）：
1. 说"你是谁"→ General 自我介绍
2. 说"你是 PM，给 TODO 写 PRD"→ 切换角色，产出 PRD
3. 说"直接开始开发"→ **被拒绝**（门禁！）
4. 说"进入 Review"→ 触发 Review 流程

> 可视化标注：CTA 按钮紫色实心醒目；体验脚本用折叠 details/summary，避免信息过载

### 第二层：组织架构层（压轴，差异化卖点）

> 目标：拉开与官方角色智能体的差异，展示"这不是单角色智能体，而是一人公司 AI 组织操作系统"。

#### 2.1 三层架构（沿用 proposal 的 layer-card）

**内容**：生态层 / 跨项目沟通层 / 项目内部层

> 可视化：直接复用 proposal 的三层色卡（紫 layer-1 / 蓝 layer-2 / 绿 layer-3），内容已验证有效

#### 2.2 指挥官—参谋长制（新增，P14 差异化）

**内容**：
P14 组织定位升级后，生态采用"指挥官—参谋长制·薄公司"形态：
- **指挥官（Owner）**：唯一决策者，亲自进组当 PM
- **参谋长**：出方案、办事务、传信息；不决策、不进组、不手改下游文件
- **公告板（coordination）**：跨项目协调仓库，字段级白名单直写
- **半自动是设计选择**：决策权下放留给下一次定位升级

> 可视化标注：组织架构图（指挥官在顶，参谋长在侧，公告板在中间，3 个项目组在底），用 proposal 的卡片风格

#### 2.3 BCR 闭环（沿用 proposal 的双通道 + 强化）

**内容**：
两条独立通道：REQUESTS 需求池 + BCR 框架变更池。BCR 状态机：已提报→评估中→已采纳→已落地真源→回流中→已回流下游。

**真实自修正记录**（沿用 proposal 的 self-evolution-box）：
- BCR-004/006：UI 角色并入 PM、Tester 角色并入 Developer 自测
- BCR-007：P14 组织定位升级（指挥官—参谋长制）
- 这不是纸面方法论，而是会自我迭代的活系统

> 可视化：复用 proposal 的 dual-channel + self-evolution-box

#### 2.4 游标机制（沿用 proposal 的 cursor-box）

**内容**：
每个下游项目有 `.workflow-version` 文件记录同步游标，防止各项目规则版本漂移。

> 可视化：复用 proposal 的 cursor-box（黄色）

## 风格约束（给 3.2 HTML 实现用）

- **配色**：深紫渐变背景 `#0f0c29 → #302b63 → #24243e`，白色内容卡片
- **字体**：`-apple-system, BlinkMacSystemFont, "PingFang SC", "Microsoft YaHei", sans-serif`
- **卡片风格**：圆角 12-20px，阴影 `0 20px 60px rgba(0,0,0,0.25)`
- **色卡**：紫 `#8b5cf6` / 蓝 `#3b82f6` / 绿 `#10b981` / 黄 `#eab308` / 粉 `#ec4899`
- **响应式**：移动端单列，桌面端网格
- **单页**：一个 HTML 文件，无外部依赖（CSS 内联），可直接打开或部署
