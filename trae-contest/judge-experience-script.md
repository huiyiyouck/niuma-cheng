# 工作流体验脚本（任务 1.7 产出）

> 配套：`workflow-demo-plan.md` 第 11 节线 1
> 场景："你是 PM，给 TODO 功能写 PRD 并走 Review"
> 目标：5 分钟跑完一轮，亲手触发门禁，感受工作流的约束力
> 验收：照脚本 5 分钟能跑完；亲手触发至少 2 处门禁

---

## 为什么这样设计

工作流的核心价值是**门禁**——不是 AI 想干什么就干什么，而是按角色、按流程、按门禁推进。这份体验脚本让你在 5 分钟内亲手触发门禁，感受"工作流不让 AI 乱来"的价值。

你会遇到的 4 个门禁点：
1. **角色门禁**：不说角色触发词，智能体保持 General，不主动加载工作流
2. **Bootstrap 门禁**：没有项目事实，智能体不写 PRD
3. **PRD Review 门禁**：PRD 写完不能直接开发，必须先走 Review
4. **迭代门禁**：没有 PRD 定稿，不能启动标准迭代开发

---

## 5 分钟体验步骤

### 准备

点智能体分享链接 → 导入"一人公司 AI 工作流"智能体到 TRAE IDE。

### 第 1 步：试探智能体身份（0:00-0:30）

**你输入**：
```
@一人公司AI工作流 你是谁
```

**预期响应**：
- 自我介绍是 General（通用助手）
- 说明可切换 4 个工作流角色：PM（产品经理）/ Architect（架构师）/ Developer（开发工程师）/ DevOps（运维部署工程师）
- 提示如何触发："你是 PM" / "进入团队工作流" 等

> 🎯 **门禁 ① · 角色门禁**：你不说角色触发词，智能体不会主动加载工作流、不读 baseline 规则、不创建 progress 文件。这是工作流的入口约束——AI 不能擅自进入工作流模式。

### 第 2 步：切换 PM 角色并下需求（0:30-1:00）

**你输入**：
```
@一人公司AI工作流 你是 PM，给一个 TODO 功能写 PRD
```

**预期响应**：
1. 切换到 PM 角色，加载工作流
2. 执行启动检查：读取项目状态
3. 如果项目未初始化 → 引导先填 3 个基本事实（项目名、定位、技术栈）
4. 如果已初始化 → 直接进入 PRD 创建

> 🎯 **门禁 ② · Bootstrap 门禁**：没有项目事实不能直接写 PRD。PM 会先确认项目上下文是否存在，不存在则引导填写。这是工作流的状态约束——AI 不能在没有项目事实的情况下凭空产出 PRD。

> 💡 提示：如果想跳过初始化直接体验 PRD 流程，可以预先在当前目录建一个 `demo-todo-project/` 并完成初始化。没建也没关系，智能体会快速引导你填 3 个事实就继续。

### 第 3 步：PM 产出 PRD（1:00-3:00）

**预期行为**：
PM 按工作流模板产出 PRD，结构大致如下：

```markdown
# v0.1 PRD

## 文档状态
- 整体状态：待Review
- Review 轮次：R1
- 关联迭代：v0.1

## 1. 迭代目标
实现一个基础 TODO 功能，支持任务创建、查看、完成标记。

## 2. 用户故事
- 作为用户，我想创建一个 TODO 任务，以便记录待办事项
- 作为用户，我想查看所有 TODO 任务，以便了解待办全貌
- 作为用户，我想标记 TODO 为已完成，以便追踪进度

## 3. 验收标准
- 当用户输入任务内容并提交时，应该创建一条新 TODO，显示在列表中
- 当用户查看任务列表时，应该看到所有未完成和已完成的任务
- 当用户点击任务的完成按钮时，该任务状态应该变为"已完成"并显示对应样式
- 当用户提交空内容时，应该提示"任务内容不能为空"（异常路径）

## 4. 范围边界
本迭代做：任务创建、列表查看、完成标记
本迭代不做：任务删除、任务编辑、优先级、截止时间、分类标签

## 5. 前置依赖
无

## 6. 风险和开放问题
- 是否需要持久化存储？还是内存即可？（待 Owner 决策）

## Review 记录
（待 Review 后填写）
```

PM 产出后会：
- 写入 `docs/progress/iterations/v0.1-prd.md`
- 更新迭代状态为"待 Review"
- 告诉你："PRD 已创建，状态为待 Review，需要指定 Review 方走 Review 流程才能定稿"

### 第 4 步：试试跳过 Review 直接开发（3:00-4:00）

**你输入**：
```
@一人公司AI工作流 PRD 写好了，直接开始开发吧
```

**预期响应**：
- **拒绝直接开发**
- 告诉你："PRD 当前状态为待 Review，按工作流门禁，必须先走 Review 流程并定稿，才能启动标准迭代开发"
- 引导你说"进入 Review"触发 Review 流程

> 🎯 **门禁 ③ · PRD Review 门禁**：PRD 写完不能直接开发，必须先走 Review。这是工作流的质量约束——AI 不能绕过 Review 直接写代码。

> 这一步是最有冲击力的体验点：你让 AI 直接写代码，它明确拒绝你。这一刻你能直观感受到——工作流不是 AI 想干啥就干啥，而是有规矩的。

### 第 5 步：触发 Review 流程（4:00-5:00）

**你输入**：
```
@一人公司AI工作流 进入 Review
```

**预期响应**：
1. 触发 Review 流程
2. 指定 Review 方（单角色可由 Owner 兼任，或指定 Architect 审设计）
3. 在 PRD 的 Review 记录区追加 Review 状态
4. 告诉你："Review 已启动，等待 Review 方反馈。Review 通过后 PRD 定稿，可启动标准迭代开发"

> 🎯 **门禁 ④ · 迭代门禁**：没有 PRD 定稿，不能启动标准迭代开发。这是工作流的流程约束——AI 不能在没有定稿 PRD 的情况下开始写代码。

### 体验结束

智能体会输出一段总结：

```
✅ 你刚完成了一轮工作流体验：
1. 角色门禁：不说角色触发词，我不进工作流
2. Bootstrap 门禁：没有项目事实，我不写 PRD
3. PRD Review 门禁：PRD 写完不能直接开发，必须先走 Review
4. 迭代门禁：没有 PRD 定稿，不能启动标准迭代

这就是工作流的核心价值：让 AI 协作从"聊天即规则"升级为"可治理的工程系统"。

想继续体验？可以说：
- "切换到 Architect，做技术设计"
- "切换到 Developer，开始开发"
- "进入团队工作流"了解更多
```

## 🔒 内部 · 彩排注意事项（不对外）

> 以下是内部彩排用的检查清单，仅供彩排和验证使用，不包含在对外物料中。

1. **演示项目预置**：建议预置 `demo-todo-project/` 已完成 Bootstrap，避免体验者卡在 Bootstrap 引导
2. **Token/字符上限验证**：如果智能体在第 3 步产出 PRD 时被截断，说明提示词超限，需回 1.1 压缩
3. **门禁拒绝措辞**：第 4 步的"拒绝直接开发"是核心体验点，提示词必须确保智能体在这里明确拒绝而不是顺从
4. **5 分钟时长**：第 3 步 PM 产出 PRD 是最耗时的，提示词要让 PM 快速产出而不反复确认

---

## 🔒 内部 · 对提示词草稿的约束（不对外）

本脚本对提示词草稿的约束：
1. 提示词必须包含角色切换路由（General→PM/Architect/Developer/DevOps）
2. 提示词必须包含门禁约束：PRD 待 Review 时拒绝直接开发、无 PRD 定稿时拒绝启动迭代
3. 提示词必须包含 PRD 模板结构（或引用 Skill 加载模板）
4. 提示词必须让 PM 在产出 PRD 时按模板快速产出，不反复确认
5. 首轮完整门禁体验后输出小结，确保体验者能看到门禁总结
